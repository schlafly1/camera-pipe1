● No need to docker rm. Here's the distinction that matters for your setup: 

docker compose stop — stops containers, keeps them. The TRT engine (which 
lives inside the container's writable layer) survives. Next docker compose up 
-d resumes in seconds.

docker compose down — stops AND removes containers. The TRT engine is gone. 
Next startup rebuilds it from the ONNX file (~5 minutes). 

So the right reboot workflow is:

# optional — Docker stops containers gracefully on shutdown anyway
docker compose -f cam1.yml stop 

# ... reboot ...

# after reboot, on host first:
xhost +local:

# bring containers back (chromadb auto-restarts due to restart:unless-stopped,
# deepstream containers need this explicit up)
docker compose -f cam1.yml up -d

# then start the pipelines as usual 
docker exec -it cam1-deepstream-cam1-1 bash # → DISPLAY=:1 python3 
pipeline2.py
docker exec -it cam1-deepstream-cam2-1 bash # → DISPLAY=:1 python3 
pipeline2.py
docker exec -it cam1-deepstream-cam1-1 bash # → python3 query_server.py

The one time you'd use docker rm is if you rebuild the image (docker compose
build) and want to force fresh containers from the new image — but even then
docker compose up -d recreates containers automatically when the image
changes, without a manual docker rm.

To start fresh:
sudo rm -rf /home/roger/sd/cam1/chroma_data/* 
rm -rf /home/roger/sd/cam1/snapshots/*

sudo is needed for chroma_data because the ChromaDB container writes those
files as root. After clearing it, ChromaDB will recreate an empty database
automatically on next startup — no container restart needed.

  ❯ i freqently get pop-up warnings on my jetson thor saying "system throttled    
  due to over-current". maybe one a minute. should i be concerned about this?   
  what to do?                                                                   
                                                                                
● Yes, you should take it seriously. "Over-current" on a Jetson Thor means the  
  system is drawing more power than the power supply can cleanly deliver, and   
  the firmware is throttling the CPU/GPU to protect itself.

  What's likely happening                                                       
   
  Running 3-4 camera pipelines simultaneously plus Gemma/Nemotron inference is a
   heavy GPU workload. The Jetson Thor has a maximum TDP in the 60-80W range,
  and your tegrastats already showed 66W draw at peak. If your power supply is  
  undersized or the cable/connector has resistance, momentary spikes trip the
  over-current protection.

  What to check first

  # See current power mode and limits                                           
  sudo nvpmodel -q
                                                                                
  # Watch power draw live while pipeline is running                             
  tegrastats | grep -o 'VIN [0-9]*mW/[0-9]*mW'                                  
                                                                                
  The VIN (total input) current reading is what matters. If it's regularly      
  hitting your supply's rated limit, you're at the edge.                        
                                                                                
  Things to do    

  1. Check your power supply — the Thor developer kit spec calls for a specific 
  supply (typically 19V, high amperage). A generic or borderline supply will
  struggle under GPU load. Verify yours matches the official spec.              
  2. Set a lower power mode to cap total draw:
  sudo nvpmodel -m 2   # try mode 2 or 3 — lower power ceiling                  
  sudo jetson_clocks --restore                                                  
  2. Run nvpmodel -q --verbose to see all available modes and their wattage     
  caps.                                                                         
  3. Stagger pipeline startup — don't start all 4 camera containers
  simultaneously. Each TRT engine build spikes power. Start them 30-60 seconds  
  apart.          
  4. Raise SAVE_INTERVAL in pipeline2.py — currently 5 seconds. Raising it to   
  10-15 reduces how often Gemma fires, which smooths out GPU power spikes.      
  5. Check the power connector — on dev kits a loose or thin barrel connector
  can cause voltage sag under load even with a good supply, which the firmware  
  reads as over-current.
                                                                                
  The throttling itself won't damage anything — that's what it's designed to    
  prevent. But if it's happening every minute it means you're regularly hitting
  the limit, and sustained operation in that state will cause dropped frames and
   slow inference. Worth fixing before adding more cameras.
